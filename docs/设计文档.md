# kraymini 设计文档

## 1. 项目概述

### 1.1 定位

kraymini 是一个 Xray 服务管理器，负责从订阅源自动拉取代理节点，生成带负载均衡和健康检查的 Xray 配置，并管理 Xray 进程的生命周期。它以持久服务的形式运行，定时刷新订阅并自动重载配置，让用户无需手动维护节点列表和 Xray 配置。

### 1.2 目标

- 从多个订阅源拉取节点，自动合并去重
- 生成完整的 Xray 运行配置，包含 observatory + leastPing 自动选优
- 支持可选的落地代理（链式出站）
- 管理 Xray 子进程的启动、监控和重载
- 定时刷新订阅，节点变更时重新生成 xray 配置并重载 xray
- 支持用户通过 kraymini 配置文件自定义路由规则和 DNS 分流
- 跨平台运行：Linux / macOS / OpenWrt

### 1.3 非目标

- 不提供 Web UI 或 HTTP API
- 不实现节点测速和排序（依赖 Xray 内置 observatory）
- 不管理 GeoIP / GeoSite 数据文件的更新（由用户手动下载所需数据文件，缺失可能导致包含 `geosite:` / `geoip:` 规则的配置校验失败）
- 不实现 Clash/Surge 等其他代理核心的支持
- 不实现流量统计和监控面板
- 不监听 kraymini 配置文件变更；修改后需重启 kraymini 生效

### 1.4 技术栈

| 项目 | 选型 |
|------|------|
| 语言 | Python 3.11+ |
| 配置格式 | TOML（标准库 `tomllib`） |
| CLI | `argparse`（标准库） |
| 进程托管 | 由外部服务管理器（如 systemd / supervisor / procd）管理 kraymini，kraymini 管理 xray |
| 外部依赖 | 尽量仅使用标准库 |

## 3. kraymini 配置设计

kraymini 使用 TOML 格式的配置文件统一管理自身参数和需要暴露给用户的 xray 配置片段。用户通过 kraymini 配置文件直接定义订阅源、端口、落地代理、路由规则和 DNS 分流；`log`、`api`、`inbounds`、`outbounds`、`routing.balancers`、`observatory` 等其余部分由 kraymini 自动生成。

### 3.1 配置文件搜索路径

kraymini 按以下顺序查找配置文件，**找到第一个即停止**：

1. CLI 参数 `-c <path>`（最高优先级）
2. `./config.toml`（当前工作目录）
3. `~/.kraymini/config.toml`
4. `/usr/local/etc/kraymini/config.toml`
5. `/etc/kraymini/config.toml`

优先级从高到低：命令行指定 > 当前目录 > 用户级 > 系统级。

kraymini 默认使用 `~/.kraymini/` 作为用户级运行时目录，用于存放默认生成的 xray 配置、节点缓存及其他运行时文件。目录不存在时自动创建。

### 3.2 配置结构

#### `[general]` 基础设置

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `xray_bin` | string | 否 | `"xray"` | xray 二进制路径，默认从 PATH 查找 |
| `output_config` | string | 否 | `"~/.kraymini/xray.json"` | 生成的 xray 配置输出路径 |
| `refresh_interval` | integer | 否 | `10800` | 订阅刷新间隔（秒） |
| `node_include` | array[string] | 否 | `[]` | 节点备注名或地址包含关键词白名单，为空表示不过滤 |
| `node_exclude` | array[string] | 否 | `[]` | 节点备注名或地址排除关键词黑名单，匹配则排除 |

路径类字段在加载时统一执行 `~` 展开并规范化为绝对路径。

`node_include` 和 `node_exclude` 对节点备注名与节点地址（URI 中的主机名或 IP）做子串匹配（不区分大小写）。同时配置时，先执行 include 过滤（仅保留匹配任一 include 关键词的节点），再执行 exclude 过滤（排除匹配任一 exclude 关键词的节点）。

#### `[[subscriptions]]` 订阅源列表

TOML 数组表，支持多个订阅源。

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `url` | string | 是 | — | 订阅地址 |
| `name` | string | 否 | `""` | 源标识名，用于日志和节点 tag 前缀 |

至少需要配置一个订阅源，否则启动时报错退出。

#### `[inbound]` 本地服务端口

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `listen` | string | 否 | `"127.0.0.1"` | 三个 inbound 共用的监听地址 |
| `socks_port` | integer | 否 | `10808` | SOCKS5 代理端口 |
| `http_port` | integer | 否 | `10809` | HTTP 代理端口 |
| `api_port` | integer | 否 | `10810` | xray API 端口（dokodemo-door） |
| `sniffing` | boolean | 否 | `true` | 是否开启流量嗅探 |

默认监听地址为 `127.0.0.1`，避免生成未鉴权的开放代理。若需监听其他地址，必须由用户显式配置。

#### `[landing_proxy]` 落地代理（可选）

不配置此段则 balancer 直接使用各订阅节点出站。配置后公网出口为落地机：经订阅节点连接落地（链式 outbound `LP-Via: <节点备注>`，见 5.3）。

基础协议字段：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `protocol` | string | 是 | — | 协议：`vmess` / `vless` / `trojan` / `shadowsocks` |
| `address` | string | 是 | — | 落地代理地址 |
| `port` | integer | 是 | — | 落地代理端口 |
| `uuid` | string | 条件 | — | vmess/vless 使用 |
| `password` | string | 条件 | — | trojan/shadowsocks 使用 |
| `method` | string | 条件 | — | shadowsocks 加密方式 |

`[landing_proxy.transport]`（可选，不配置则使用协议默认传输）：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `network` | string | 否 | `"tcp"` | 传输方式：`tcp` / `ws` / `grpc` / `h2` |

`[landing_proxy.transport.ws]`（当 `network = "ws"` 时可用）：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `path` | string | 否 | `"/"` | WebSocket path |
| `host` | string | 否 | `""` | WebSocket `Host` 头 |

`[landing_proxy.transport.grpc]`（当 `network = "grpc"` 时可用）：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `service_name` | string | 否 | `""` | gRPC serviceName |
| `multi_mode` | boolean | 否 | `true` | 是否启用 gRPC 多路复用 |

`[landing_proxy.transport.h2]`（当 `network = "h2"` 时可用）：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `path` | string | 否 | `"/"` | HTTP/2 path |
| `host` | array[string] | 否 | `[]` | HTTP/2 Host 列表 |

`[landing_proxy.security]`（可选，不配置则使用 `none`）：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `mode` | string | 否 | `"none"` | 安全层：`none` / `tls` / `reality` |
| `server_name` | string | 否 | `""` | TLS/Reality 的 SNI / serverName |
| `allow_insecure` | boolean | 否 | `false` | 是否跳过证书校验 |
| `fingerprint` | string | 否 | `"chrome"` | TLS/Reality 指纹伪装 |
| `alpn` | array[string] | 否 | 自动推导 | ALPN 列表；为空时按传输方式自动补全 |

`[landing_proxy.security.reality]`（当 `mode = "reality"` 时必填）：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `public_key` | string | 是 | — | Reality 公钥 |
| `short_id` | string | 否 | `""` | Reality shortId |
| `spider_x` | string | 否 | `"/"` | Reality spiderX |

为避免 TLS `server_name` 与 WebSocket/HTTP2 `Host` 混淆，安全层字段与传输层字段分开定义，不再复用单个 `host` 字段表达多个语义。

#### `[routing]` 路由规则（可选）

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `domain_strategy` | string | 否 | `"IPOnDemand"` | 域名解析策略：`AsIs` / `IPIfNonMatch` / `IPOnDemand` |
| `domain_matcher` | string | 否 | `"mph"` | 域名匹配算法：`linear` / `mph` |

`[[routing.rules]]` 为 TOML 数组表，每条规则对应一条 xray routing rule（`type` 固定为 `"field"`，由 kraymini 自动填充）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `domain` | array[string] | 否 | 域名匹配列表，支持 `geosite:` 前缀 |
| `ip` | array[string] | 否 | IP 匹配列表，支持 CIDR 和 `geoip:` 前缀 |
| `network` | string | 否 | 网络类型：`tcp` / `udp` / `tcp,udp` |
| `inbound_tag` | array[string] | 否 | 来源 inbound tag 列表 |
| `outbound_tag` | string | 是 | 目标 outbound tag |

可用的 `outbound_tag` 值：`direct`、`blocked`；配置了 `[landing_proxy]` 时还可使用 `LP-Via: <节点备注>`（与自动生成的链式出站 tag 一致）。不需要也不允许引用裸订阅节点 tag、`api` 或 balancer；API 规则和 balancer 兜底规则均由 kraymini 自动生成。

不配置 `[routing]` 时，kraymini 仅生成 API 规则和 balancer 兜底规则，所有流量走代理。

#### `[dns]` DNS 配置（可选）

`[dns.hosts]` 为静态域名映射表（`string → string`），对应 xray `dns.hosts`。

`[[dns.servers]]` 为 DNS 服务器列表，所有服务器均使用对象形式（不支持纯字符串形式，但可只填 `address`）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `address` | string | 是 | DNS 服务器地址，支持 IP、`https://` DoH、`localhost` |
| `port` | integer | 否 | 端口号，默认 53 |
| `domains` | array[string] | 否 | 该服务器负责解析的域名列表，支持 `geosite:` 前缀 |
| `expect_ips` | array[string] | 否 | 期望返回的 IP 范围，不匹配则丢弃，支持 `geoip:` 前缀 |

不配置 `[dns]` 时，xray 配置中不生成 `dns` 字段，xray 使用系统默认 DNS。

#### `[observatory]` 健康检查

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `probe_url` | string | 否 | `"https://www.google.com/generate_204"` | 探测目标 URL |
| `probe_interval` | string | 否 | `"5m"` | 探测间隔（xray duration 格式，如 `5m`、`30s`） |
| `enable_concurrency` | boolean | 否 | `true` | 是否并发探测全部匹配节点；节点较多时建议保持开启 |

负载均衡策略固定为 `leastPing`，与 observatory 配合实现自动选优。

#### `[log]` 日志

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `level` | string | 否 | `"info"` | kraymini 日志级别：`debug` / `info` / `warning` / `error` |
| `xray_level` | string | 否 | `"warning"` | xray 日志级别：`debug` / `info` / `warning` / `error` / `none` |
| `file` | string | 否 | `""` | 日志文件路径，为空则 kraymini 输出到 stdout；非空则 kraymini 与 xray 共用该文件 |

kraymini 日志格式：`<时间戳> <级别> <内容>`，示例：`2026-04-11 10:30:00 INFO 订阅刷新完成，共 42 个节点`。

### 3.3 默认值策略

kraymini 的设计原则是**最小配置可运行**。一个最简配置只需要：

```toml
[[subscriptions]]
url = "https://example.com/sub?token=xxx"
```

其余字段全部使用默认值。此时 kraymini 会：
- 从 PATH 查找 xray 二进制
- 自动创建 `~/.kraymini/` 运行时目录
- 在 `127.0.0.1:10808`（SOCKS）/ `127.0.0.1:10809`（HTTP）/ `127.0.0.1:10810`（API）启动服务
- 将生成的 xray 配置写入 `~/.kraymini/xray.json`
- 不使用落地代理
- 不配置路由规则和 DNS（生成仅含 inbound + outbound + balancer + observatory 的最简 xray 配置，所有流量走代理）
- 将节点缓存写入 `~/.kraymini/`
- 每 10800 秒（3 小时）刷新一次订阅

### 3.4 配置校验

kraymini 启动时执行以下校验，不通过则报错退出：

- 至少存在一个 `[[subscriptions]]` 且 `url` 非空
- 端口值在 1-65535 范围内，且三个端口互不冲突
- `[inbound].listen` 如指定，必须是合法的 IPv4 或 IPv6 地址（如 `127.0.0.1`、`0.0.0.0`、`::1`、`::`）
- `[landing_proxy]` 如存在，`protocol` / `address` / `port` 必填
- `[landing_proxy]` 的凭证字段按协议校验：`vmess` / `vless` 需要 `uuid`；`trojan` 需要 `password`；`shadowsocks` 需要 `password + method`
- `[landing_proxy.transport]` 如存在，`network` 必须属于 `tcp` / `ws` / `grpc` / `h2`
- 当 `network = "ws"` 时，仅使用 `[landing_proxy.transport.ws]`；当 `network = "grpc"` 时，仅使用 `[landing_proxy.transport.grpc]`；当 `network = "h2"` 时，仅使用 `[landing_proxy.transport.h2]`
- `[landing_proxy.security].mode = "reality"` 时，必须提供 `server_name` 和 `[landing_proxy.security.reality].public_key`
- `[[routing.rules]]` 中必须指定 `outbound_tag`，且其值必须是 `direct`、`blocked` 之一；若配置了 `[landing_proxy]`，还可为 `LP-Via: <节点备注>`（与链式出站 tag 一致）
- `[[routing.rules]]` 每条规则至少指定 `domain`、`ip`、`network`、`inbound_tag` 之一
- `[[dns.servers]]` 中 `address` 为必填
- `xray_bin` 对应的二进制（默认从 PATH 查找 `xray`，或用户指定的路径）必须存在且可执行
- `refresh_interval` 必须为正整数
- `[log].level` 必须属于 `debug` / `info` / `warning` / `error`
- `[log].xray_level` 必须属于 `debug` / `info` / `warning` / `error` / `none`
- `[routing].domain_strategy` 如指定，必须属于 `AsIs` / `IPIfNonMatch` / `IPOnDemand`
- `[routing].domain_matcher` 如指定，必须属于 `linear` / `mph`
- 默认运行时目录 `~/.kraymini/` 不存在时自动创建；创建失败则报错退出

同机多实例部署时，必须显式配置不同的 `output_config`、端口和日志路径，避免互相覆盖。

### 3.5 完整配置示例

```toml
[general]
xray_bin = "/usr/local/bin/xray"
output_config = "~/.kraymini/xray.json"
refresh_interval = 10800
node_exclude = ["剩余流量", "到期时间", "套餐"]

[[subscriptions]]
url = "https://provider-a.com/sub?token=xxx"
name = "provider-a"

[[subscriptions]]
url = "https://provider-b.com/sub?token=yyy"
name = "provider-b"

[inbound]
listen = "127.0.0.1"
socks_port = 10808
http_port = 10809
api_port = 10810
sniffing = true

[landing_proxy]
protocol = "trojan"
address = "landing.example.com"
port = 443
password = "your-password"

[landing_proxy.transport]
network = "tcp"

[landing_proxy.security]
mode = "tls"
server_name = "landing.example.com"
fingerprint = "chrome"

[observatory]
probe_url = "https://www.google.com/generate_204"
probe_interval = "5m"

[routing]
domain_strategy = "IPOnDemand"
domain_matcher = "mph"

[[routing.rules]]
domain = [
    "baidu.com", "taobao.com", "aliyun.com", "alidns.com",
    "weixin.com", "tencent.com", "qq.com",
    "geosite:cn"
]
outbound_tag = "direct"

[[routing.rules]]
ip = ["119.29.29.29", "223.5.5.5", "geoip:private", "geoip:cn"]
outbound_tag = "direct"

[[routing.rules]]
domain = ["geosite:category-ads"]
outbound_tag = "blocked"

[dns]
[dns.hosts]
"mysite" = "donothing.site"

[[dns.servers]]
address = "119.29.29.29"
port = 53
domains = ["geosite:cn"]
expect_ips = ["geoip:cn"]

[[dns.servers]]
address = "8.8.8.8"
port = 53
domains = ["geosite:geolocation-!cn"]

[[dns.servers]]
address = "1.1.1.1"

[log]
level = "info"
xray_level = "warning"
file = "~/.kraymini/kraymini.log"
```

## 4. 订阅管理

### 4.1 订阅拉取流程

```
HTTP GET 订阅 URL（多源并行）
    → Base64 解码响应体
    → 按行分割，得到节点 URI 列表
    → 逐条解析 URI 为内部节点数据结构
    → 多源合并 + 去重
    → 日志逐条记录每个节点信息（协议类型、节点名称、地址、端口、网络传输协议，不输出密码/UUID 等敏感信息）
    → 按关键词过滤（如配置了 node_include / node_exclude）
    → 日志记录过滤结果汇总（保留 N 个，排除 M 个）
    → 缓存到本地
```

多个订阅源使用线程池并行拉取（`concurrent.futures.ThreadPoolExecutor`），所有成功的源合并节点列表后统一去重。

### 4.2 支持的协议及 URI 格式

| 协议 | URI 格式 | 备注名来源 |
|------|---------|-----------|
| VMess | `vmess://` + Base64 JSON | JSON 中的 `ps` 字段 |
| VLESS | `vless://uuid@host:port?params#remark` | URL fragment |
| Trojan | `trojan://password@host:port?params#remark` | URL fragment |
| Shadowsocks | `ss://base64@host:port#remark` 或 `ss://base64#remark` | URL fragment |
| Hysteria2 | `hy2://password@host:port?params#remark` | URL fragment |

#### 解析器扩展机制

`parser/` 目录下每个协议一个模块，每个模块需导出一个 `parse(uri: str) -> Node` 函数。`parser/__init__.py` 维护协议前缀到解析函数的注册表：

```python
PARSERS: dict[str, Callable[[str], Node]] = {
    "vmess://": vmess.parse,
    "vless://": vless.parse,
    "trojan://": trojan.parse,
    "ss://": shadowsocks.parse,
    "hy2://": hysteria.parse,
}
```

新增协议步骤：
1. 在 `parser/` 下新建模块（如 `tuic.py`），实现 `parse(uri: str) -> Node`
2. 在 `PARSERS` 注册表中添加 `"tuic://": tuic.parse`
3. 在 `generator.py` 的 outbound 生成逻辑中添加该协议到 xray outbound JSON 的映射

每种协议的 URI 解析为统一的内部数据结构：

```python
@dataclass
class Node:
    raw_uri: str          # 原始 URI，用于去重
    remark: str           # 节点备注名
    protocol: str         # vmess / vless / trojan / ss / hysteria2
    address: str          # 服务器地址
    port: int             # 服务器端口
    credentials: dict     # 协议特定凭证（uuid / password / method 等）
    transport: dict       # 传输层参数（network / security / path / host 等）
    source: str           # 来源订阅名（用于日志）
```

### 4.3 节点命名

xray outbound 的 `tag` 使用节点自带的备注名。命名处理按以下顺序执行：

1. **空备注名兜底**：备注名为空时，使用 `<source_name>-<序号>` 生成名称（如 `provider-a-0`）；`source_name` 也为空时，使用 `sub` 作为默认前缀（如 `sub-0`）
2. **保留 tag 冲突处理**：节点名称与内部保留 tag 冲突时，自动追加 `_node` 后缀
3. **全局去重**：对所有名称（包括经过上述处理的）统一检查重复，重名节点追加数字后缀

```
香港-01          ← 第一个
香港-01_2        ← 重名的第二个
美国-高速        ← 无重名，原样使用
direct_node      ← 原名 direct，与保留 tag 冲突
direct_node_2    ← 另一个也叫 direct 的节点，去重后追加后缀
```

kraymini 内部保留或系统占用的名称：

- `LP-Via: ` 前缀：配置了 `[landing_proxy]` 时自动生成的链式出站 tag 为 `LP-Via: <节点备注>`
- `direct`
- `blocked`
- `api`
- `balancer`
- `in-api`、`in-socks`、`in-http`

### 4.4 去重策略

以节点的 **完整原始 URI** 作为去重键。相同 URI 出现在多个订阅源时，保留第一个遇到的，后续丢弃。

选择完整 URI 而非 `address + port` 的原因：同一地址端口可能对应不同用户、不同传输配置。

### 4.5 缓存

每次成功拉取并解析后，将合并去重过滤后的节点列表序列化缓存到 `~/.kraymini/` 下。

- 缓存文件名统一为 `nodes-cache-<hash8>.json`（对配置文件绝对路径取 SHA-256 前 8 位十六进制），存放在 `~/.kraymini/` 下，天然支持多实例隔离
- 运行时目录建议权限为 `0700`，缓存文件权限为 `0600`
- 缓存包含节点凭证和传输参数，仅用于本机恢复，不应共享给其他用户或主机

缓存仅在订阅拉取失败时作为回退使用。使用缓存时记录 WARNING 日志，注明回退原因和缓存文件路径。

加载缓存时捕获 JSON 解析异常，缓存文件损坏（格式错误、数据不完整）视同缓存不存在，记录 WARNING。

### 4.6 失败处理

| 场景 | 行为 |
|------|------|
| 部分订阅源拉取失败 | 使用成功的源继续，日志记录 WARNING |
| 全部订阅源拉取失败且本地有缓存 | 回退到本地缓存并记录 WARNING；后续按正常流程比较节点变化并决定是否重载 |
| 全部订阅源拉取失败且无缓存（首次启动） | 记录 CRITICAL，报错退出（exit code 2） |
| 全部订阅源拉取失败且无缓存（运行中刷新） | 记录 ERROR，保持当前 xray 运行不变，等待下次刷新 |
| 拉取成功但解析后节点数为 0 且有缓存 | 视同全部失败，回退缓存，记录 WARNING |
| 拉取成功但解析后节点数为 0 且无缓存（首次启动） | 记录 CRITICAL，报错退出（exit code 2） |
| 拉取成功但解析后节点数为 0 且无缓存（运行中刷新） | 记录 ERROR，保持当前 xray 运行不变，等待下次刷新 |

### 4.7 拉取参数

| 参数 | 值 |
|------|-----|
| 超时时间 | 10 秒 |
| 重试次数 | 3 次 |
| 重试间隔 | 指数退避，基数 1 秒（1s → 2s → 4s） |
| User-Agent | 模拟常见客户端 UA，避免被订阅商屏蔽 |

## 5. Xray 配置生成

### 5.1 生成流程

```
读取 kraymini 配置
    → 拉取订阅，失败时回退本地缓存
    → 生成 inbounds
    → 生成 outbounds（节点 + 若有落地则每节点一条 LP-Via 链 + direct + blocked）
    → 生成 balancer + observatory
    → 生成 routing（API 规则 + 用户规则 + balancer 兜底规则）
    → 生成 dns（如配置了 [dns]）
    → 生成 log / api / stats / policy
    → 组装完整 xray JSON 配置
    → 写入临时文件（<output_config>.tmp）
    → 调用 xray run -test -c <临时文件> 校验
    → 校验通过：重命名临时文件为 output_config
    → 校验失败：删除临时文件，保留旧配置
```

### 5.2 inbounds 生成

根据 `[inbound]` 配置固定生成三个 inbound：

| tag | 协议 | 地址来源 | 端口来源 | 说明 |
|-----|------|---------|---------|------|
| `in-api` | dokodemo-door | `listen` | `api_port` | xray API |
| `in-socks` | socks | `listen` | `socks_port` | SOCKS5 代理 |
| `in-http` | http | `listen` | `http_port` | HTTP 代理 |

`in-socks` 和 `in-http` 根据 `sniffing` 配置决定是否开启流量嗅探（`destOverride: ["http", "tls"]`）。

默认 `listen = "127.0.0.1"`，避免生成未鉴权开放代理。若用户需要监听其他地址，必须显式修改配置。

### 5.3 outbounds 生成

outbounds 按以下顺序排列：

1. **落地链式 outbound**（如配置了 `[landing_proxy]`）— 每个节点先生成一条，tag 为 `LP-Via: <节点备注>`，协议与 `[landing_proxy]` 一致，且 `proxySettings` 指向对应节点 outbound（见下「落地代理挂载」）
2. **订阅节点 outbound** — 每个节点生成一个，tag 使用节点备注名（见 4.3）
3. **`direct` outbound** — 固定生成，protocol `freedom`，`domainStrategy: "UseIP"`
4. **`blocked` outbound** — 固定生成，protocol `blackhole`

#### 节点 outbound 生成规则

从节点的 `Node` 数据结构转换为 xray outbound JSON。URI 中能解析出的参数直接使用，缺失的传输层参数按以下社区推荐值补全：

**TLS 默认值（当 `security` 为 `tls` 时）：**

| 字段 | 默认值 | 说明 |
|------|--------|------|
| `allowInsecure` | `false` | 不跳过证书验证 |
| `fingerprint` | `"chrome"` | TLS 指纹伪装 |
| `alpn` | 根据传输方式自动选择 | ws/tcp → `["http/1.1"]`，grpc/h2 → `["h2"]` |

**WebSocket 默认值（当 `network` 为 `ws` 时）：**

| 字段 | 默认值 |
|------|--------|
| `headers.User-Agent` | 模拟 Chrome 浏览器 UA |
| `headers.Accept-Encoding` | `"gzip, deflate, br"` |
| `headers.Accept-Language` | `"zh-CN,zh;q=0.9"` |

**gRPC 默认值（当 `network` 为 `grpc` 时）：**
- `multiMode` 默认 `true`（开启多路复用）

#### 落地代理挂载

配置了 `[landing_proxy]` 时：

- **订阅节点** outbound **不**再添加 `proxySettings`（仅作为链的底层，用于连接落地机）。
- 对每个节点，**额外**生成一条与 `[landing_proxy]` 协议相同的 outbound，tag 为 `LP-Via: <节点备注>`，并添加：

```json
{
  "tag": "LP-Via: 节点备注名",
  "protocol": "trojan",
  "settings": { ... },
  "streamSettings": { ... },
  "proxySettings": {
    "tag": "节点备注名",
    "transportLayer": true
  }
}
```

即：**连接落地服务器**时先走对应 **节点 outbound**；访问公网时出口为**落地机 IP**。落地 outbound 的字段由 `[landing_proxy]`、`[landing_proxy.transport.*]` 和 `[landing_proxy.security.*]` 映射生成。

流量链路：`客户端 → 订阅节点（第一跳） → 落地代理（第二跳，公网出口） → 目标`。

`balancer` 与 `observatory` 的 selector / subjectSelector 使用所有以 `LP-Via: ` 开头的 tag，与探测路径一致。

为避免 `leastPing` 在启动初期或所有链式节点暂时无健康数据时回退到默认 outbound 并绕过落地代理，配置了 `[landing_proxy]` 时：

- `balancer.fallbackTag` 固定生成为 `blocked`
- 链式 outbound 排在订阅节点 outbound 之前，作为额外的 fail-close 防线

`direct` 和 `blocked` outbound 不挂载落地代理。

### 5.4 routing 生成

routing 各字段来源：

- `domainStrategy`：取自 `[routing].domain_strategy`，默认 `"IPOnDemand"`
- `domainMatcher`：取自 `[routing].domain_matcher`，默认 `"mph"`
- `rules`：按以下顺序拼接
- `balancers`：由 kraymini 生成，用户无需配置

**rules 拼接顺序（固定）：**

1. **API 规则**（kraymini 自动插入最前）：

```json
{ "type": "field", "inboundTag": ["in-api"], "outboundTag": "api" }
```

2. **用户规则**（来自 `[[routing.rules]]`，按配置顺序排列）

3. **balancer 兜底规则**（kraymini 自动追加最后）：

```json
{ "type": "field", "network": "tcp,udp", "balancerTag": "balancer" }
```

**balancer**：

```json
{
  "tag": "balancer",
  "selector": ["节点tag-1", "节点tag-2", ...],
  "fallbackTag": "blocked",
  "strategy": { "type": "leastPing" }
}
```

未配置 `[landing_proxy]` 时，`selector` 为所有订阅节点 tag，且不生成 `fallbackTag`；配置了 `[landing_proxy]` 时，`selector` 为所有 `LP-Via: <节点备注>` tag，并额外生成 `fallbackTag: "blocked"`，以避免无健康节点时流量落到裸订阅节点。负载均衡策略固定为 `leastPing`，与 observatory 配合实现自动选优。

未配置 `[routing]` 时，rules 仅含 API 规则和 balancer 兜底规则两条。

### 5.5 observatory 生成

```json
{
  "subjectSelector": ["节点tag-1", "节点tag-2", ...],
  "probeURL": "https://www.google.com/generate_204",
  "probeInterval": "5m",
  "enableConcurrency": true
}
```

`probeURL`、`probeInterval` 和 `enableConcurrency` 取自 `[observatory]` 配置。`subjectSelector` 与 balancer 的 `selector` 保持一致。

### 5.6 API 配置

固定生成，用于支持 observatory 和管理：

```json
{
  "tag": "api",
  "services": [
    "HandlerService",
    "LoggerService",
    "RoutingService",
    "ObservatoryService"
  ]
}
```

### 5.7 stats 与 policy 配置

固定生成，为 observatory 和 leastPing 策略提供必要的统计基础：

**stats**（启用统计模块）：

```json
{}
```

**policy**（启用出站流量统计）：

```json
{
  "system": {
    "statsOutboundUplink": true,
    "statsOutboundDownlink": true
  }
}
```

`stats` 为空对象即可激活统计模块；`policy.system` 开启出站上下行流量统计，observatory 依赖此数据进行节点健康评估和 leastPing 选优。

### 5.8 输出字段汇总

生成的 xray 配置各顶层字段来源：

| 顶层字段 | 来源 |
|---------|------|
| `log` | `[log].xray_level` → `loglevel`；不配置 `access` 和 `error` 文件路径，xray 日志输出到标准流由 kraymini 统一管理（见 6.3） |
| `api` | kraymini 固定生成 |
| `inbounds` | 由 `[inbound]` 生成 |
| `outbounds` | 由（若配置 `[landing_proxy]` 则先生成每节点一条 `LP-Via: …` 链式出站）+ 订阅节点 + 固定的 `direct` / `blocked` 生成 |
| `routing.domainStrategy` | `[routing].domain_strategy`，默认 `"IPOnDemand"` |
| `routing.domainMatcher` | `[routing].domain_matcher`，默认 `"mph"` |
| `routing.rules` | API 规则 + `[[routing.rules]]` 用户规则 + balancer 兜底规则 |
| `routing.balancers` | 由订阅节点或以 `LP-Via: ` 为前缀的 tag 列表生成（取决于是否配置落地），strategy 固定为 `leastPing`；配置落地时额外生成 `fallbackTag: "blocked"` |
| `observatory` | 由 `[observatory]` 生成 |
| `dns` | 由 `[dns]` 生成；未配置则不生成 |
| `stats` | kraymini 固定生成（空对象 `{}`，启用统计模块） |
| `policy` | kraymini 固定生成（见 5.7 节） |

### 5.9 配置校验

生成 JSON 后，先写入临时文件（如 `<output_config>.tmp`），然后调用 `xray run -test -c <临时文件路径>` 校验合法性。

| 校验结果 | 行为 |
|---------|------|
| 通过 | 将临时文件重命名并覆盖至 `output_config` 路径 |
| 失败 | 删除临时文件，不覆盖旧配置，日志记录 ERROR 并说明校验失败原因及附上 xray 的错误输出 |

## 6. 进程管理与调度

### 6.1 进程管理架构

```
外部服务管理器（可选，如 systemd / supervisor / procd）
  └── kraymini（持久服务，长驻运行）
        └── xray（子进程，由 kraymini 启动和管理）
```

外部服务管理器只负责守护 kraymini 进程，并不是 kraymini 的运行前提。kraymini 负责管理 xray 子进程的完整生命周期：启动、监控、重载、停止。

### 6.2 启动流程

```
kraymini 启动
  → 加载并校验配置
  → 拉取订阅
  → 拉取成功：生成 xray 配置 + xray run -test 校验 → 启动 xray 子进程
  → 拉取失败：读取本地缓存 → 生成 xray 配置 + xray run -test 校验 → 启动 xray 子进程
  → 配置校验失败：记录 ERROR，以 exit code 2 退出
  → 进入主循环（定时刷新 + 子进程监控）
```

启动时订阅拉取全部失败且本地无缓存，kraymini 报错退出（无法生成有效配置）。

### 6.3 xray 子进程管理

kraymini 通过 `subprocess.Popen` 启动 xray：

```
<xray_bin> run -c <output_config>
```

并持有子进程引用，持续监控其状态。

| 操作 | 实现方式 |
|------|---------|
| 启动 | `subprocess.Popen([xray_bin, "run", "-c", config_path])` |
| 停止 | 发送 `SIGTERM`，等待 5 秒，超时则 `SIGKILL` |
| 重载 | 停止旧进程 → 启动新进程（使用新配置） |
| 状态检查 | `process.poll()` 检查进程是否存活 |

#### 子进程日志输出

xray 配置中不设置 `log.access` 和 `log.error` 文件路径时，xray 将访问/错误日志写到标准流（通常为 stderr）。kraymini 通过 `Popen` 管理 xray 的日志输出：

- `[log].file` 非空时：kraymini 打开该文件，将 xray 子进程的 stderr 重定向到此文件；kraymini 自身也写入同一文件
- `[log].file` 为空时：kraymini 日志写入 **stdout**；xray 子进程继承父进程标准流，其日志通常仍在 **stderr**，与 kraymini 的 stdout 分离

#### 重载流程

```
检测到需要重载（订阅节点变更）
  → xray run -test 校验新配置
  → 校验通过：SIGTERM 停止旧 xray → 启动新 xray
  → 校验失败：保留旧进程和旧配置，日志记录 ERROR
```

重载期间存在短暂的服务中断（旧进程停止到新进程就绪之间）。这在实际使用中通常只有毫秒级，但**会导致当前已建立的所有 TCP/UDP 连接（如 SSH、下载等）强行断开**。因此 `refresh_interval` 默认设置为 3 小时，可根据需要调整刷新频率。

### 6.4 子进程异常处理

kraymini 主循环中持续监控 xray 进程状态：

| 场景 | 行为 |
|------|------|
| xray 进程意外退出 | 等待 3 秒后自动重启，日志记录 ERROR |
| 连续崩溃（30 秒内崩溃 3 次） | 停止重启尝试，日志记录 CRITICAL，kraymini 继续运行；下次订阅刷新时重置崩溃计数并尝试重新启动 xray（无论节点是否变化） |
| kraymini 收到 SIGTERM/SIGINT | 先停止 xray 子进程，再退出自身 |

### 6.5 定时刷新调度

kraymini 主循环中按 `refresh_interval` 周期执行刷新：

```
等待 refresh_interval 秒
  → 拉取订阅
  → 拉取失败：回退到本地缓存
  → 缓存也无可用节点：记录 ERROR，保持当前 xray 运行不变，等待下次刷新
  → 与当前节点列表比对
  → 有变更：重新生成配置 → 重载 xray
  → 无变更：跳过，继续等待
```

变更判定：将新节点列表的 URI 集合与当前集合比较，有差异则视为变更。

kraymini **只**监听订阅刷新，不监听 kraymini 配置文件变化。配置文件修改后，需要重启 kraymini 才会生效。

### 6.6 主循环伪代码

```python
POLL_INTERVAL = 5
CRASH_WINDOW = 30
MAX_CRASHES = 3
force_refresh = False

def on_sighup(signum, frame):
    global force_refresh
    force_refresh = True

signal.signal(signal.SIGHUP, on_sighup)

def main_loop():
    global force_refresh
    config = load_config()
    nodes = fetch_or_load_cache(config)
    xray_config = generate_xray_config(config, nodes)
    xray_proc = start_xray(xray_config)
    last_refresh = time.time()
    crash_times: list[float] = []
    crash_suspended = False

    while True:
        sleep(POLL_INTERVAL)

        if not crash_suspended and xray_proc.poll() is not None:
            now = time.time()
            crash_times = [t for t in crash_times if now - t < CRASH_WINDOW]
            crash_times.append(now)

            if len(crash_times) >= MAX_CRASHES:
                log.critical("xray 连续崩溃 %d 次，暂停重启", MAX_CRASHES)
                crash_suspended = True
            else:
                log.error("xray 意外退出，3 秒后重启")
                sleep(3)
                xray_proc = start_xray(xray_config)

        need_refresh = force_refresh or (time.time() - last_refresh >= config.refresh_interval)
        if not need_refresh:
            continue

        force_refresh = False
        last_refresh = time.time()
        new_nodes = fetch_or_load_cache(config)
        if new_nodes is None:
            log.error("订阅拉取失败且缓存不可用，保持当前配置继续运行")
            continue

        if nodes_changed(nodes, new_nodes) or crash_suspended:
            nodes = new_nodes
            new_config = generate_xray_config(config, nodes)
            if validate_xray_config(new_config):
                xray_proc = reload_xray(xray_proc, new_config)
                xray_config = new_config
                crash_times.clear()
                crash_suspended = False
```

### 6.7 信号处理

| 信号 | 行为 |
|------|------|
| `SIGTERM` | 优雅关闭：停止 xray 子进程 → kraymini 退出 |
| `SIGINT` | 同 SIGTERM |
| `SIGHUP` | 立即触发一次订阅刷新（不等待下一个刷新周期） |

### 6.8 supervisor 配置示例（部署示例）

完整示例见仓库内 `deploy/kraymini.supervisor.conf`。核心段落如下（`redirect_stderr=false`：kraymini 主日志在 stdout，xray 诊断多在 stderr，故分文件收集；若设为 `true` 则 stderr 并入 stdout，不必再配 `stderr_logfile`）：

```ini
[program:kraymini]
command=/usr/local/bin/kraymini run -c /etc/kraymini/config.toml
directory=/var/lib/kraymini
user=nobody
autostart=true
autorestart=true
startsecs=5
startretries=3
stopwaitsecs=10
stopasgroup=true
killasgroup=true
redirect_stderr=false
stdout_logfile=/var/log/kraymini/supervisor.log
stdout_logfile_maxbytes=20MB
stdout_logfile_backups=5
stderr_logfile=/var/log/kraymini/supervisor.err.log
stderr_logfile_maxbytes=10MB
stderr_logfile_backups=3
environment=PATH="/usr/local/bin:/usr/bin:/bin"
```

`stopwaitsecs` 设为 10 秒，给 kraymini 足够时间停止 xray 子进程后再退出。若运行在 systemd 或 OpenWrt procd 下，可按各自服务管理器的方式做等价配置。

## 7. CLI 设计

### 7.1 命令风格

kraymini 使用子命令风格的 CLI 接口：

```
kraymini <command> [options]
```

### 7.2 子命令列表

#### `kraymini run`

长驻运行模式。拉取订阅、生成配置、启动 xray、进入主循环。

```
kraymini run [-c <config_path>]
```

| 参数 | 说明 |
|------|------|
| `-c, --config` | 指定配置文件路径，不指定则按搜索路径查找 |

这是 kraymini 的主命令，通常由外部服务管理器或用户手动启动。

#### `kraymini genconfig`

仅生成 xray 配置文件，不启动 xray 进程。用于调试和验证。

```
kraymini genconfig [-c <config_path>] [-o <output_path>]
```

| 参数 | 说明 |
|------|------|
| `-c, --config` | 指定 kraymini 配置文件路径 |
| `-o, --output` | 输出路径，不指定则使用配置中的 `output_config`；指定为 `-` 则输出到 stdout |
| `--offline` | 跳过订阅拉取，仅使用本地缓存生成配置。缓存不存在时报错退出 |

流程：加载配置 → 拉取订阅（失败时回退缓存；`--offline` 时直接使用缓存，缓存不存在则报错退出） → 生成 xray 配置 → `xray run -test` 校验 → 输出。不启动 xray，执行完即退出。

#### `kraymini check`

校验 kraymini 配置文件的合法性。

```
kraymini check [-c <config_path>]
```

校验内容同 3.4 节。通过则输出 `OK` 并以 exit code 0 退出，失败则输出错误信息并以 exit code 1 退出。

#### `kraymini version`

输出版本信息。

```
kraymini version
```

输出格式（版本号与源码中 `kraymini.__version__` 一致，打包元数据由该值同步）：

```
kraymini <version>
```

### 7.3 全局参数

| 参数 | 说明 |
|------|------|
| `-c, --config` | 所有子命令共享，指定配置文件路径 |
| `-v, --verbose` | 将 kraymini 日志级别提升至 DEBUG（覆盖配置文件中的 `level`，不影响 `xray_level`） |
| `-h, --help` | 显示帮助信息 |

### 7.4 退出码

| 退出码 | 含义 |
|-------|------|
| 0 | 正常退出 |
| 1 | 配置错误（解析失败、校验不通过） |
| 2 | 运行时错误（订阅全部失败且无缓存、xray 二进制不存在等） |

### 7.5 实现

- 使用 Python 标准库 `argparse` 实现，不引入额外 CLI 框架依赖。
- **网络请求超时**：使用 `urllib.request` 拉取订阅时，需显式设置 `timeout` 参数（如 `timeout=10`），避免网络阻塞导致线程卡死。
- **TOML 解析异常捕获**：使用 `tomllib` 解析时，需捕获 `tomllib.TOMLDecodeError`，并提供友好的配置文件格式错误提示，避免抛出复杂的 Traceback。
- **跨平台信号处理**：对于 `subprocess.Popen`，`SIGTERM` 在 Windows 上行为有所不同，但项目主要针对 Linux / macOS / OpenWrt 等 POSIX 环境设计，可忽略 Windows 的信号处理差异。

## 8. 验收与测试矩阵

### 8.1 测试层次

| 层次 | 目标 | 典型对象 |
|------|------|---------|
| 单元测试 | 校验纯逻辑与边界条件 | 配置解析、URI 解析、去重、命名、路由规则映射、DNS 配置映射 |
| 集成测试 | 校验模块协作 | 订阅拉取 + 缓存、生成配置 + `xray run -test`、进程重载 |
| 端到端验收 | 校验可交付行为 | `run` / `genconfig` / `check` 命令、定时刷新、崩溃恢复 |

集成测试中建议使用本地 mock HTTP 服务模拟订阅源，使用可替换的假 xray 二进制模拟 `run -test` 和运行态进程，避免依赖真实网络环境。

### 8.2 功能验收矩阵

| 编号 | 场景 | 类型 | 预期结果 |
|------|------|------|---------|
| A01 | 仅配置一个订阅 URL 的最简配置启动 | 端到端 | 自动创建 `~/.kraymini/`，生成 `xray.json`，默认监听 `127.0.0.1`，成功启动 xray |
| A02 | 配置文件搜索路径优先级 | 单元/集成 | `-c` > `./config.toml` > `~/.kraymini/config.toml` > `/usr/local/etc/kraymini/config.toml` > `/etc/kraymini/config.toml` |
| A03 | 非法端口、冲突端口、非法监听地址 | 单元 | `check` 失败并返回 exit code 1 |
| A04 | `landing_proxy` 缺少协议必填字段 | 单元 | 配置校验失败并给出明确错误信息 |
| A05 | `landing_proxy` 的 `ws` / `grpc` / `h2` / `tls` / `reality` 配置映射 | 单元/集成 | 生成的 outbound JSON 与配置语义一致，`server_name` 与各类 `Host` 字段不混淆 |
| A06 | 多订阅源合并与按原始 URI 去重 | 单元 | 相同 URI 仅保留首个节点，不同 URI 即使地址端口相同也保留 |
| A07 | 节点重名、空备注、保留 tag 冲突 | 单元 | 按顺序执行兜底命名、保留 tag 冲突处理、全局去重 |
| A07a | `node_include` / `node_exclude` 关键词过滤 | 单元 | include 为空时保留全部；对备注与地址子串匹配；exclude 排除匹配节点；两者同时配置时先 include 后 exclude |
| A08 | 部分订阅源失败 | 集成 | 使用成功源继续生成节点并记录 WARNING |
| A09 | 全部订阅失败但本地有缓存 | 集成 | 回退到缓存节点并继续生成配置；若结果无变化则不重载，有变化则通过校验后重载 |
| A10 | 首次启动全部订阅失败且无缓存 | 端到端 | 启动失败并返回运行时错误 exit code 2 |
| A11 | `genconfig --offline` 且缓存存在 | 端到端 | 仅使用 `~/.kraymini/` 缓存生成并校验配置，不访问网络 |
| A12 | `[routing]` / `[dns]` 配置映射 | 集成 | 生成的 `routing` / `dns` JSON 与 TOML 配置语义一致，自动插入 API 规则和 balancer 兜底规则 |
| A13 | `[[routing.rules]]` 非法 `outbound_tag` | 单元/集成 | 配置校验失败，并给出明确错误信息 |
| A14 | 节点集合无变化的定时刷新 | 集成 | 不重写配置、不重载 xray |
| A15 | 节点集合有变化的定时刷新 | 集成 | 重新生成配置，通过校验后重载 xray |
| A15a | 运行中刷新失败且缓存不可用 | 集成 | 记录 ERROR，保持当前 xray 运行不变，不中断服务 |
| A16 | xray 子进程异常退出 | 集成 | 3 秒后自动重启；达到连续崩溃阈值后暂停重启并记录 CRITICAL；下次刷新时重置崩溃计数并重启 |
| A17 | `SIGTERM` / `SIGINT` 终止 | 端到端 | 优雅停止 xray 后退出 kraymini |
| A18 | `SIGHUP` 立即刷新 | 集成 | 收到 SIGHUP 后立即触发订阅拉取，不等待刷新周期 |
| A19 | 配置校验失败时临时文件处理 | 集成 | 生成配置校验失败后，临时文件被删除，旧 `output_config` 保持不变 |
| A20 | 节点解析日志输出 | 单元 | 每个节点记录协议、名称、地址、端口、传输协议；不输出 UUID/密码等敏感字段 |
